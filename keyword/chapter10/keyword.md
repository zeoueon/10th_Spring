# 클라우드 컴퓨팅이란?

## ☁️ 클라우드 컴퓨팅이란?
서버, 스토리지, 데이터베이스, 네트워크 등의 IT 자원을 **물리적으로 직접 소유하지 않고**, 인터넷을 통해 필요한 만큼 빌려 쓰고 사용한 만큼만 비용을 지불하는 컴퓨팅 방식이다.

기존 On-Premise 방식을 사용하면 기업은 직접 인프라를 구축해 IT 서비스를 개발해야한다. 이때 기업은 자체 비즈니스에 집중하지 못하고 인프라 관리에 보다 더 많은 신경을 써야한다. 따라서 요즘 많은 기업은 비용을 지불하고 클라우드 서비스를 사용하는 방식을 많이 채택하고있다.

대표적인 **클라우드 제공 업체(CSP, Cloud Service Provider)**로는 **AWS, Microsoft Azure, Google Cloud Platform(GCP)** 등이 있다.

---

## 온프레미스(On-Premise)와 클라우드

**온프레미스 (On-Premise)**
자원을 직접 소유해서 운영하는 방식이며, 그만큼 서버를 구매하고 구축하는 데에 초기 비용이 많이든다. 서버의 확장이 필요할 때 직접 물리 장비를 추가해야하기 때문에 확장성이 떨어질 수 있다.

**클라우드 (Cloud)**
클라우드 제공 업체의 서비스를 빌려서 사용한는 방식이며, 초기 비용이 많이들지 않는다. 서버의 확장이 필요할 때 서비스 사용량을 늘리는 것과 같이 즉시 확장이 가능하다. 또한 CSP가 관리에 대한 책임을 지고 있어 기업 입장에서 인프라 관리 부담을 많이 덜 수 있다. 

---

## 클라우드의 핵심 특징

**1️⃣ 주문형 셀프 서비스 (On-demand Self-service)**
필요한 자원을 사람의 개입 없이 콘솔이나 API로 즉시 받아 쓸 수 있다.

**2️⃣ 신속한 탄력성 (Elasticity)**
트래픽이 몰리면 자원을 자동으로 늘리고(Scale-out), 줄어들면 다시 줄여 비용을 최적화할 수 있다.

**3️⃣ 사용량 기반 과금 (Pay-as-you-go)**
사용한 자원만큼만 비용을 지불한다. 유휴 자원에 대한 낭비가 적다.

**4️⃣ 자원 풀링 (Resource Pooling)**
물리 자원을 가상화하여 여러 사용자가 공유한다. 사용자는 물리적 위치를 신경 쓸 필요가 없다.

---

## 클라우드가 제공하는 자원의 종류

클라우드는 서비스 운영에 필요한 거의 모든 IT 자원을 가상화하여 제공한다. 크게 아래 네 가지로 나눌 수 있다.

**1️⃣ 컴퓨팅 (Computing)**
연산을 수행하는 핵심 자원이다. CPU·메모리를 가진 가상 서버를 빌려 애플리케이션을 실행한다.
- 예시: **AWS EC2**(가상 서버), AWS Lambda(서버리스), GCP Compute Engine

**2️⃣ 스토리지 (Storage)**
데이터를 저장하는 자원이다. 저장 방식에 따라 종류가 나뉜다.
- **객체 스토리지(Object)**: 이미지·동영상·백업 등 파일 저장 → **AWS S3**
- **블록 스토리지(Block)**: 서버에 디스크처럼 붙이는 저장소 → **AWS EBS**
- **파일 스토리지(File)**: 여러 서버가 공유하는 파일 시스템 → AWS EFS

**3️⃣ 데이터베이스 (Database)**
데이터를 구조화하여 관리하는 자원이다. CSP가 설치·백업·장애 복구를 대신 관리해준다.
- **관계형(RDB)**: **AWS RDS** (MySQL, PostgreSQL 등)
- **NoSQL**: AWS DynamoDB
- **인메모리/캐시**: AWS ElastiCache (Redis)

**4️⃣ 네트워크 (Network)**
자원들을 연결하고 외부 트래픽을 분배·보호하는 자원이다.
- **가상 네트워크**: AWS VPC (격리된 사설 네트워크 구성)
- **로드 밸런서**: AWS ELB (트래픽을 여러 서버로 분산)
- **CDN**: AWS CloudFront (전 세계 캐싱으로 응답 속도 향상)

## 클라우드 서비스 모델 (IaaS / PaaS / SaaS)

제공자(CSP)가 **어디까지 관리해주느냐**에 따라 책임 범위가 나뉜다. 

```text
IaaS  →  PaaS  →  SaaS
(자유도 높음)   (편의성 높음)

[ 직접 관리 영역 ]        [ CSP 관리 영역 ]
IaaS : 앱 / 런타임 / OS    | 가상화 / 서버 / 네트워크
PaaS : 앱 / 데이터          | 런타임 / OS / 인프라 전부
SaaS : (설정·사용만)        | 애플리케이션까지 전부
```

### IaaS (Infrastructure as a Service)
서버, 스토리지, 네트워크 같은 **인프라 자원만** 빌려주는 모델이다. CSP는 가상화·물리 서버·네트워크까지만 책임지고, 그 위의 **OS·런타임·애플리케이션은 사용자가 직접 설치하고 운영**한다.

기존 시스템을 그대로 클라우드로 옮기거나, 인프라를 직접 통제하고 싶을 때 사용한다.

- **장점**: 자유도가 가장 높다. OS 종류, 서버 사양, 네트워크 구성을 원하는 대로 세팅할 수 있어 세밀한 제어가 필요한 경우에 적합하다.
- **단점**: 직접 관리해야 하는 영역이 많아 운영 부담과 인프라 전문 지식이 요구된다.
- **AWS EC2**, AWS S3, GCP Compute Engine

### PaaS (Platform as a Service)
애플리케이션을 **개발·배포할 수 있는 플랫폼**을 제공한다. OS·런타임·미들웨어·인프라까지 CSP가 모두 관리하므로, 개발자는 **오직 코드와 데이터에만 집중**할 수 있다.

인프라 관리 없이 빠르게 서비스를 개발 및 배포하고 싶을 때 사용한다.

- **장점**: 인프라를 신경 쓸 필요가 없어 개발 생산성이 높다. 배포·확장도 플랫폼이 알아서 처리해준다.
- **단점**: 제공되는 플랫폼의 환경·언어·버전에 맞춰야 하므로 IaaS보다 자유도가 낮다.
- **AWS Elastic Beanstalk**, Heroku, Google App Engine

### SaaS (Software as a Service)
완성된 **소프트웨어 자체를 서비스로** 제공한다. 사용자는 설치·관리 없이 인터넷(브라우저)으로 바로 사용한다. 개발 영역이 아니라 **최종 사용자가 바로 쓰는** 형태다.

- **장점**: 설치·유지보수가 전혀 필요 없고 구독만 하면 즉시 사용 가능하다.
- **단점**: 정해진 기능 안에서만 사용 가능해 커스터마이징이 가장 제한적이다.
- **Gmail, Notion, Google Docs, Dropbox**

### 한눈에 비교

| 구분        | IaaS          | PaaS                      | SaaS          |
| --------- | ------------- | ------------------------- | ------------- |
| 제공 단위     | 인프라 (서버·스토리지) | 개발 플랫폼                    | 완성된 소프트웨어     |
| 사용자 관리 범위 | OS 위 전부       | 앱·데이터만                    | 설정·사용만        |
| 자유도       | 높음            | 중간                        | 낮음            |
| 주 사용자     | 인프라 관리자       | 개발자                       | 최종 사용자        |
| 예시        | EC2, S3       | Elastic Beanstalk, Heroku | Gmail, Notion |

## 클라우드 배포 모델 (Public / Private / Hybrid)

서비스 모델(IaaS·PaaS·SaaS)이 **무엇을 빌리느냐**의 구분이라면, 배포 모델은 클라우드 인프라를 **누가, 어떻게 소유하고 사용하느냐**에 대한 구분이다.

### Public Cloud
AWS·Azure·GCP 같은 CSP가 구축한 인프라를 **불특정 다수가 공용으로 나눠 쓰는** 방식이다. 우리가 흔히 말하는 클라우드가 대부분 이것이다.

- **장점**: 초기 구축 비용이 없고, 사용한 만큼만 지불한다. 자원을 즉시 확장할 수 있다.
- **단점**: 인프라를 직접 통제할 수 없고, 규제가 엄격한 데이터(금융·의료 등)는 외부에 두기 부담스러울 수 있다.
- 예시: AWS, Microsoft Azure, GCP

### Private Cloud
**특정 조직(기업) 전용으로** 구축해 독점적으로 사용하는 클라우드다. 자체 데이터센터에 두거나 전용 공간을 임대해 운영한다.

- **장점**: 인프라를 완전히 통제할 수 있어 **보안·규제 준수**에 유리하다. 민감 데이터를 외부에 노출하지 않는다.
- **단점**: 직접 구축·운영해야 하므로 초기 비용과 인프라 관리 부담 절감의 장점을 누리기 힘들다. 
- **언제 쓰나**: 금융·의료·공공기관처럼 데이터 보안과 규제 준수가 핵심일 때.

**온프레미스 vs 프라이빗 클라우드**
둘 다 특정 조직 전용 인프라라는 점은 같지만, **물리 서버를 쓰는 방식**이 다르다.

- **온프레미스**: 물리 서버 1대를 그냥 **컴퓨터 1대로** 사용한다. (서버 1대 = 용도 1개)
- **프라이빗 클라우드**: 그 물리 서버를 **가상화**해서 여러 OS(가상 컴퓨터)를 띄워두고, **콘솔/API로 "1번 가상 서버를 쓰겠다"는 식으로** 필요한 만큼 골라 제어한다.

### Hybrid Cloud 
**Public + Private을 함께** 사용하는 방식이다. 두 모델의 장점을 상황에 맞게 조합한다.

민감한 데이터·핵심 시스템은 **Private**에 두어 보안을 지키고, 그 외 일반 트래픽이나 급증하는 부하는 **Public**으로 처리해 비용과 확장성을 챙긴다.

---

## 왜 클라우드를 쓰는가

- **비용 효율**: 초기 인프라 투자 없이 사용한 만큼만 지불한다.
- **확장성**: 트래픽 급증 시 즉시 자원을 늘릴 수 있어 안정적인 서비스 운영이 가능하다.
- **빠른 출시**: 인프라 구축에 드는 시간을 절약해 서비스를 빠르게 배포할 수 있다.
- **가용성·안정성**: CSP가 여러 지역(Region)·가용 영역(AZ)에 자원을 분산해 장애에 대응한다.

-> 이런 장점이 있지만, 장기적인 관점에서 봤을 때 서비스가 클라우드에 종속되고 온프레미스 방식보다 비용이 더 발생할 수 있다는 점을 고려해서 운영해야한다. 

# AWS와 GCP

## AWS, GCP란?
둘 다 인터넷을 통해 서버·스토리지·데이터베이스 등의 IT 자원을 빌려주는 대표적인 **퍼블릭 클라우드 제공 업체(CSP, Cloud Service Provider)**다. 우리가 클라우드 서비스를 쓴다고 하면 보통 이런 CSP의 인프라를 빌려 사용하는 것을 의미한다.

- **AWS (Amazon Web Services)**: Amazon이 2006년에 출시한 **시장 점유율 1위**의 클라우드다. 가장 먼저 등장한 클라우드 서비스이고, 제공하는 서비스가 가장 많다.
- **GCP (Google Cloud Platform)**: Google이 운영하는 클라우드로, **데이터·AI/ML, 쿠버네티스(컨테이너)** 분야에 강점이 있다.
- 이외에 Microsoft의 **Azure**가 점유율 2위로, 보통 이 셋을 **3대 CSP**라고 부른다.

---

## AWS (Amazon Web Services)

가장 오래되고 규모가 큰 클라우드라 **서비스 종류와 자료·커뮤니티가 압도적으로 많다.** 처음 클라우드를 입문하는 사람들이 사용하기에 좋다.

### 대표 서비스
- **EC2** (Elastic Compute Cloud): 가상 서버 (컴퓨팅)
- **S3** (Simple Storage Service): 객체 스토리지 (이미지·파일·백업)
- **RDS** (Relational Database Service): 관리형 관계형 DB
- **Lambda**: 서버 없이 코드만 실행하는 서버리스
- **VPC** (Virtual Private Cloud): 격리된 가상 네트워크

### 특징
- **장점**: 서비스가 가장 많고 생태계가 잘 구축되어있다. 자료·커뮤니티가 풍부해서 처음 공부하기에 좋고, 문제해결도 쉽다.
- **단점**: 서비스가 너무 많아 처음엔 복잡하게 느껴질 수 있고, 콘솔/요금 구조가 다소 어렵다.

---

## GCP (Google Cloud Platform)

Google의 인프라(검색·유튜브를 돌리는 그 기반) 위에서 동작한다. **데이터 분석, AI/ML, 컨테이너(쿠버네티스)** 영역에서 특히 강점을 보인다. 쿠버네티스(Kubernetes) 자체가 구글에서 만든 기술이다.

### 대표 서비스
- **Compute Engine**: 가상 서버 (EC2에 대응)
- **Cloud Storage**: 객체 스토리지 (S3에 대응)
- **Cloud SQL**: 관리형 관계형 DB (RDS에 대응)
- **BigQuery**: 대용량 데이터 분석 (GCP의 간판 서비스)
- **GKE** (Google Kubernetes Engine): 관리형 쿠버네티스

### 특징
- **장점**: 데이터 분석(BigQuery)·AI/ML·쿠버네티스가 강력하다. 콘솔이 비교적 깔끔하고 직관적이며, 과금이 합리적이라는 평이 많다.
- **단점**: AWS보다 서비스 종류와 레퍼런스(자료)가 적어, 국내 채용 시장에서는 AWS 경험을 더 많이 요구한다.

## 🔐 환경변수(Environment Variable)란?
애플리케이션 코드 **바깥(운영체제·실행 환경)**에 저장해두고, 프로그램이 실행될 때 가져다 쓰는 값이다. DB 비밀번호, API 키, 외부 서비스 시크릿처럼 **코드에 직접 박으면 안 되는 값**을 분리해서 관리하는 데 주로 쓰인다.

코드에 값을 직접 적는 대신(**하드코딩**), 환경변수로 빼두면 같은 코드를 환경(개발/운영)만 바꿔서 재사용할 수 있고, 민감 정보가 코드와 함께 노출되는 것을 막을 수 있다.

## 왜 민감 정보를 환경변수로 가려야 하는가

**1️⃣ 소스코드 유출 = 비밀 유출**
코드에 비밀번호·API 키를 직접 적으면, 코드가 노출되는 순간 비밀도 함께 털린다. 특히 **GitHub 같은 원격 저장소에 그대로 올라가면** 누구나 볼 수 있게 된다.

**2️⃣ Git 이력에는 영원히 남는다**
한 번 커밋된 비밀번호는 나중에 코드에서 지워도 **Git 커밋 히스토리에 그대로 남는다.** 과거 커밋을 들춰보면 복구할 수 있어, 단순히 지운다고 해결되지 않는다. (이 경우 키 자체를 폐기·재발급해야 한다)

**3️⃣ 봇이 자동으로 긁어간다**
GitHub에 올라온 API 키·토큰을 **자동으로 스캐닝하는 봇**이 상시 돌아다닌다. 실수로 AWS 키를 커밋했다가 몇 분 만에 탈취되어 요금 폭탄을 맞는 사례가 흔하다.

**4️⃣ 환경별로 값이 다르다**
개발 DB와 운영 DB의 접속 정보는 다르다. 코드에 박아두면 환경마다 코드를 바꿔야 하지만, 환경변수로 빼면 **같은 코드 + 다른 환경변수**로 운영할 수 있다.

---

## 환경변수 처리 방법

### 1. `.env` 파일 사용
프로젝트 루트에 `.env` 파일을 두고 `KEY=VALUE` 형태로 값을 적어두는 방식이다. 가장 보편적이다.

```text
# .env
DB_PASSWORD=mypassword1234
JWT_SECRET=abcd1234efgh5678
AWS_ACCESS_KEY=AKIA...
```

⚠️ **반드시 `.gitignore`에 `.env`를 추가**해서 Git에 올라가지 않게 해야 한다. 이게 빠지면 환경변수를 쓰는 의미가 없어진다.

```text
# .gitignore
.env
```

### 2. `.env.example` 로 형식만 공유
`.env`는 올리지 않되, **어떤 키가 필요한지 형식만 알려주는** 예시 파일은 올린다. 팀원이 받아서 값만 채워 쓰면 된다.

```text
# .env.example  (값은 비워둠 → Git에 올려도 안전)
DB_PASSWORD=
JWT_SECRET=
AWS_ACCESS_KEY=
```

### 3. OS·서버 환경변수 등록
운영 서버에서는 파일 대신 **OS 자체의 환경변수**로 등록하는 경우가 많다.

```text
export DB_PASSWORD=mypassword1234
```

### 4. 클라우드 시크릿 매니저 (실무 권장)
규모가 커지면 `.env` 파일조차 서버에 두기 부담스러워, **전용 비밀 저장소 서비스**를 쓴다. `.env`는 결국 **평문 파일이 서버에 그대로 존재**하므로 서버가 털리면 바로 노출되고, 누가 언제 봤는지 추적도 안 되며, 키를 바꾸려면 모든 서버를 일일이 손봐야 하는 한계가 있다. CSP 시크릿 매니저는 이를 해결해준다.

---

## CSP가 제공하는 환경변수·시크릿 관리 기능

### `.env` 대비 장점
- **암호화 저장**: 값이 암호화되어 저장된다. 저장소가 털려도 평문이 바로 노출되지 않는다.
- **세밀한 접근 제어(IAM)**: "이 서버/이 역할(Role)만 이 시크릿을 읽을 수 있다"처럼 권한을 지정한다.
- **감사 로그(Audit)**: 누가 언제 어떤 시크릿에 접근했는지 전부 기록되어 추적이 가능하다.
- **자동 교체(Rotation)**: DB 비밀번호 등을 주기적으로 자동 변경해 노출 위험을 줄인다.
- **중앙 관리**: 서버가 수십~수백 대여도 한 곳에서 관리하고, 값을 바꾸면 전체에 반영된다.

### AWS

**1️⃣ Secrets Manager**
DB 비밀번호·API 키 같은 시크릿 전용 서비스다. **자동 교체(Rotation)**를 기본 지원한다. RDS와 연동하면 DB 비밀번호를 주기적으로 자동 변경해준다. (유료, 시크릿당 과금)

**2️⃣ Systems Manager Parameter Store**
설정값·시크릿을 저장하는 서비스다. 일반 설정값은 무료(Standard)이고, 민감 값은 `SecureString` 타입으로 암호화 저장한다. 자동 교체는 없지만 **비용 부담이 적어** 가벼운 프로젝트에 많이 쓴다.

-> 자동 교체·DB 연동이 필요하면 **Secrets Manager**, 단순 저장이면 **Parameter Store**를 쓰는 식으로 활용할 수 있다.

### GCP

**Secret Manager**
GCP의 시크릿 전용 서비스다. 버전 관리(시크릿 값마다 버전을 매겨 롤백 가능), IAM 기반 접근 제어, 감사 로그를 제공한다.

---

## 실제 배포 환경에서의 활용 흐름

핵심은 **애플리케이션이 시크릿을 "파일이나 코드로 들고 있지 않고", 실행 시점에 시크릿 매니저에서 꺼내 쓰는 것**이다.

```text
1. 시크릿 매니저에 값 저장
   (DB_PASSWORD, JWT_SECRET 등을 콘솔/CLI로 등록)
        ↓
2. 서버(EC2 등)에 IAM Role 부여
   "이 서버는 해당 시크릿을 읽을 수 있다"는 권한만 연결
        ↓
3. 앱이 시작될 때 시크릿 매니저 API 호출
   → 권한이 있으니 비밀번호 없이도 시크릿을 받아옴
        ↓
4. 받아온 값을 환경변수처럼 메모리에 올려 사용
```

### 활용 포인트
- **서버에 비밀번호를 두지 않는다**: 서버에는 `.env`도, 키 파일도 없다. "이 서버가 시크릿을 읽을 권한이 있다"는 **IAM Role**만 부여하면, 서버는 별도 자격 증명 없이도 시크릿을 꺼내 쓸 수 있다. 
- **CI/CD 파이프라인 연동**: GitHub Actions 등에서 배포할 때, 빌드/배포에 필요한 시크릿을 매니저에서 주입한다. 파이프라인 스크립트에도 비밀번호가 남지 않는다.
- **컨테이너(쿠버네티스) 환경**: 시크릿 매니저의 값을 Pod의 환경변수나 볼륨으로 주입해, 컨테이너 이미지 안에 비밀이 박히지 않게 한다.
- **환경별 분리**: dev/prod 환경마다 다른 시크릿을 두고, 각 환경의 서버는 자기 환경 시크릿만 읽도록 권한을 제한한다.

# YML 환경 분리 (Spring Profiles)

## ⚙️ YML 환경 분리란?
하나의 애플리케이션을 **개발(dev), 운영(prod) 등 서로 다른 환경**에서 돌릴 때, 환경마다 달라지는 설정값(DB 주소, 포트, 로그 레벨 등)을 **프로필(Profile)별로 나눠서 관리**하는 방법이다.

같은 코드라도 환경마다 접속하는 DB나 설정이 다르기 때문에, 설정을 분리해두면 **코드 수정 없이 실행 시점에 환경만 골라** 적용할 수 있다.

대표적인 프로필은 **`dev`(개발) · `test`(테스트) · `prod`(운영)** 세 가지다. 테스트 환경도 운영·개발과 분리해서 **테스트 전용 DB·설정**을 따로 두는 것이 일반적이다.

---

## 왜 환경을 분리해야할까?

**1️⃣ 환경마다 설정값이 다르다**
개발용 DB와 운영용 DB의 주소·계정이 다르다. 분리하지 않으면 배포할 때마다 설정을 직접 바꿔야 한다.

**2️⃣ 실수로 운영 환경을 건드리는 사고 방지**
개발 중에 운영 DB에 연결되어 데이터가 망가지는 사고를 막을 수 있다.

**3️⃣ 환경별 최적화**
개발에서는 상세 로그·자동 DDL을, 운영에서는 최소 로그·DDL 검증만 하도록 환경마다 다르게 둘 수 있다.

---

## 방법 1. 단일 파일에서 `---`로 분리

`application.yml` 한 파일 안에서 `---` 구분자로 여러 프로필을 나눈다. 소규모 프로젝트에서 간단하다.

```yaml
# 공통 설정 (모든 프로필에 적용)
spring:
  application:
    name: my-app
  profiles:
    active: dev      # 기본으로 활성화할 프로필

---
# dev 프로필
spring:
  config:
    activate:
      on-profile: dev
  datasource:
    url: jdbc:mysql://localhost:3306/dev_db
    username: dev_user
    password: ${DB_PASSWORD}
logging:
  level:
    root: DEBUG

---
# prod 프로필
spring:
  config:
    activate:
      on-profile: prod
  datasource:
    url: jdbc:mysql://prod-server:3306/prod_db
    username: prod_user
    password: ${DB_PASSWORD}
logging:
  level:
    root: WARN
```

> 맨 위의 `---` 이전 블록은 **공통 설정**, 이후 블록은 각 프로필 전용 설정이다. 같은 키가 있으면 활성화된 프로필 값이 공통 값을 덮어쓴다.

---

## 방법 2. 파일을 나눠서 분리

설정이 많아지면 파일 자체를 프로필별로 쪼개는 것이 깔끔하다. **이름 규칙은 `application-{프로필}.yml`** 이다.

```text
src/main/resources/
├── application.yml          # 공통 설정 + 활성 프로필 지정
├── application-dev.yml      # 개발 환경 전용
├── application-test.yml     # 테스트 환경 전용
└── application-prod.yml     # 운영 환경 전용
```

```yaml
# application.yml (공통)
spring:
  profiles:
    active: dev      # 어떤 프로필을 켤지 지정
```

```yaml
# application-dev.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/dev_db
    username: dev_user
    password: ${DB_PASSWORD}
logging:
  level:
    root: DEBUG
```

```yaml
# application-prod.yml
spring:
  datasource:
    url: jdbc:mysql://prod-server:3306/prod_db
    username: prod_user
    password: ${DB_PASSWORD}
logging:
  level:
    root: WARN
```

Spring Boot가 `application.yml`을 먼저 읽고, 활성 프로필에 맞는 `application-{프로필}.yml`을 추가로 읽어 **합쳐서** 적용한다.

---

## 프로필을 활성화하는 방법

`application.yml`에 고정으로 적을 수도 있지만, 실제 배포에서는 **실행 시점에 외부에서 지정**하는 경우가 많다. 우선순위가 더 높아 코드 수정 없이 환경을 바꿀 수 있다.

```text
# 1. 실행 인자로 지정
java -jar app.jar --spring.profiles.active=prod

# 2. 환경변수로 지정 (배포 환경에서 흔함)
export SPRING_PROFILES_ACTIVE=prod

# 3. JVM 옵션으로 지정
java -Dspring.profiles.active=prod -jar app.jar
```


---

## 테스트 환경(test) 분리

테스트 코드는 **실제 개발·운영 DB를 건드리면 안 된다.** 테스트 도중 데이터가 변경되거나, 운영 DB에 잘못 연결되는 사고를 막아야 하기 때문이다. 그래서 테스트는 보통 **`test` 프로필 + 인메모리 DB(H2)**로 격리해서 돌린다.

### application-test.yml 예시

테스트가 끝나면 사라지는 **인메모리 H2 DB**를 써서, 매 테스트가 깨끗한 상태에서 독립적으로 실행되도록 한다.

```yaml
# application-test.yml
spring:
  datasource:
    url: jdbc:h2:mem:testdb       # 메모리에만 존재 → 테스트 끝나면 소멸
    driver-class-name: org.h2.Driver
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: create-drop       # 시작 시 테이블 생성, 종료 시 삭제
    database-platform: org.hibernate.dialect.H2Dialect
logging:
  level:
    root: INFO
```

### 테스트 코드에서 프로필 지정 — `@ActiveProfiles`

테스트 클래스에 `@ActiveProfiles("test")`를 붙이면, 해당 테스트는 **`test` 프로필 설정으로** 실행된다.

```java
@SpringBootTest
@ActiveProfiles("test")   // application-test.yml 적용
class MemberServiceTest {

    @Test
    void 회원_가입_테스트() {
        // 운영 DB가 아닌 H2 인메모리 DB에서 실행됨
    }
}
```


### 테스트 환경을 따로 두는 이유
- **데이터 격리**: 운영·개발 DB를 오염시키지 않는다.
- **독립성·반복성**: 인메모리 DB는 매번 초기화되므로, 테스트가 항상 같은 결과를 보장한다.
- **속도**: 메모리에서 동작해 실제 DB보다 빠르다. CI 파이프라인에서 특히 유리하다.

## 환경 분리 + 민감정보 처리 함께 쓰기

환경 분리(YML)와 민감정보 보호(환경변수)는 **함께 쓰는 것**이다.

- **YML 환경 분리**: 환경마다 다른 *구조·설정값*을 나눔 (DB 주소, 로그 레벨 등)
- **환경변수/시크릿**: 그중 *민감한 값*(비밀번호, 키)은 `${ }`로 빼서 외부에서 주입

```yaml
# application-prod.yml — 구조는 YML로, 민감값은 환경변수로
spring:
  datasource:
    url: jdbc:mysql://prod-server:3306/prod_db   # 환경별 설정
    password: ${DB_PASSWORD}                      # 민감값은 외부 주입
```

# Docker와 .jar vs Docker 이미지

## 🐳 Docker란?
애플리케이션을 **실행에 필요한 모든 것(코드 + 라이브러리 + 런타임 + 설정)과 함께 하나로 포장**해서, 어떤 환경에서도 똑같이 실행되게 해주는 **컨테이너 기술**이다.

가장 큰 목적은 **"내 컴퓨터에선 되는데 서버에선 안 돼요" 문제를 없애는 것**이다. 실행 환경을 통째로 포장하기 때문에, 개발 PC든 운영 서버든 동일하게 동작한다.


---

## 컨테이너 vs 가상머신(VM)

Docker를 이해하려면 기존의 가상머신과 비교하면 쉽다.

| 구분    | 가상머신 (VM)          | 컨테이너 (Docker)      |
| ----- | ------------------ | ------------------ |
| 격리 단위 | OS 전체를 통째로 가상화     | 앱 + 실행 환경만 격리      |
| 무게    | 무겁다 (OS 포함, GB 단위) | 가볍다 (OS 공유, MB 단위) |
| 부팅 속도 | 느림 (수십 초~분)        | 빠름 (수 초)           |
| OS    | 게스트 OS를 따로 설치      | 호스트 OS 커널 공유       |

---

## 핵심 개념: 이미지 vs 컨테이너

Docker에서 가장 헷갈리는 두 단어다. 관계만 잡으면 쉽다.

- **이미지(Image)**: 실행에 필요한 것을 모아 만든 **설계도·템플릿** (정적, 변하지 않음)
- **컨테이너(Container)**: 그 이미지를 **실제로 실행한 인스턴스** (동적, 살아 움직임)

```text
Dockerfile  ──(build)──▶  이미지(Image)  ──(run)──▶  컨테이너(Container)
 (레시피)                  (붕어빵 틀)               (실제 붕어빵)
```

> **이미지 1개로 컨테이너 여러 개**를 띄울 수 있다. 붕어빵 틀(이미지) 하나로 붕어빵(컨테이너)을 여러 개 찍어내는 것과 같다.

## .jar vs Docker 이미지

Spring Boot 프로젝트를 빌드하면 `.jar` 파일이 나온다. 그런데 왜 굳이 Docker 이미지로 또 감쌀까? 둘이 **무엇을 포함하는지**가 핵심 차이다.

### .jar 파일
- 컴파일된 **애플리케이션 코드 + 라이브러리(의존성)**를 하나로 묶은 실행 파일이다.
- 실행하려면 **외부에 Java(JRE/JDK)가 미리 설치**되어 있어야 한다. (`java -jar app.jar`)
- 즉, 실행 환경(Java 버전, OS 설정 등)은 **그 서버에 의존**한다.

### Docker 이미지
- `.jar`은 물론 **그것을 실행할 Java 런타임, OS 환경, 설정까지 전부 포함**한다.
- 그래서 Java가 깔려있지 않은 서버에서도 그대로 실행된다. 환경이 통째로 들어있기 때문이다.

```dockerfile
# Dockerfile 예시 — .jar을 이미지로 감싸기
FROM openjdk:17-jdk-slim     # ① Java 17 실행 환경 포함
COPY build/libs/app.jar app.jar   # ② 빌드된 .jar 복사
ENTRYPOINT ["java", "-jar", "app.jar"]   # ③ 컨테이너 시작 시 실행
```

### 한눈에 비교

| 구분 | .jar | Docker 이미지 |
| --- | --- | --- |
| 포함 범위 | 앱 코드 + 라이브러리 | 앱 + 라이브러리 + **Java 런타임 + OS 환경** |
| 실행 조건 | 서버에 Java 설치 필요 | Docker만 있으면 됨 |
| 환경 일관성 | 서버 환경에 의존 (버전 충돌 가능) | 어디서나 동일하게 실행 |
| 배포 | 서버마다 Java 세팅 필요 | 이미지만 받아서 바로 실행 |
| 크기 | 작음 (수십 MB) | 큼 (런타임·OS 포함) |

## 왜 .jar를 Docker 이미지로 감싸는가

- **환경 일관성**: 개발·테스트·운영이 똑같은 환경에서 실행된다. Java 버전 차이로 인한 사고가 없다.
- **배포 간편화**: 서버에 Java를 깔 필요 없이 **이미지만 받아서 실행**(`docker run`)하면 된다.
- **확장·이식성**: 어떤 클라우드(AWS·GCP)나 쿠버네티스 환경에도 같은 이미지를 그대로 올릴 수 있다.
- **버전 관리**: 이미지에 태그(`app:1.0`, `app:2.0`)를 붙여 버전별로 관리하고 롤백할 수 있다.

```text

  │ (빌드: gradle build)
  ▼

  │ (docker build, Dockerfile 기반)
  ▼

  │ (docker run)
  ▼
