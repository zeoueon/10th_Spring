## REST란? (Representational State Transfer)

- API 작동 방식에 대한 조건을 부여하는 소프트웨어 아키텍처
- **웹에서 자원을 다루는 아키텍처 스타일(설계 원칙)**
    
    → 자원(Resource)을 URL로 표현하고, HTTP 메서드로 조작한다.
    
- REST 기반 아키텍처를 사용하여 대규모의 고성능 통신을 안정적으로 지원할 수 있다.
- 쉽게 구현하고 수정할 수 있어 모든 API 시스템을 파악하고 여러 플랫폼에서 사용할 수 있다.

## REST 아키텍처 스타일의 원칙

### 1️⃣ 균일한 인터페이스

- 모든 API는 일관된 방식으로 자원을 다뤄야한다.
    1. 자원 식별(URI): 모든 자원은 고유한 URI로 표현되어야 한다.
    2. 표현을 통한 상태 전달: 서버는 JSON과 같은 형태로 데이터를 제공한다.
    3. 자기 설명 메시지: 요청/응답만 보고도 의미를 이해 가능해야한다. 헤더, 상태코드, 메타데이터를 포함한다.
    4. 하이퍼링크 제공: 다음 행동을 링크로 알려준다. (클라이언트가 API 구조를 몰라도 되지만, 보통 모든 API의 구조를 클라이언트가 이미 알고 있다.)

### 2️⃣ 무상태

- 서버는 이전 요청을 저장하지 않는 무상태로 동작한다.
- 모든 요청은 독립적이며, 필요한 정보는 요청에 모두 포함된다.
- 무상태로 동작하기 때문에 서버 확장이 쉽고 유지 보수에 용이하다.
- ex) 모든 요청에 인증 포함 `Authorization Bearer XX`

### 3️⃣ 클라이언트 - 서버 구조

- 클라이언트와 서버의 역할을 분리하여 독립적인 개발과 확장이 가능하도록 한다.
- 클라이언트는 사용자 인터페이스를 담당하고, 서버는 비즈니스 로직과 데이터 저장을 관리한다.

### 4️⃣ 계층화 시스템

- 서버는 여러 계층으로 구성될 수 있으며, 클라이언트는 최종 서버인지 프록시 서버인지 알 필요가 없다.
- 보안과 확장성을 향상시킬 수 있다.

### 5️⃣ 캐시 가능성

- 클라이언트는 일부 응답을 캐싱할 수 있으며, 성능을 향상시킨다.

### 6️⃣ 온디맨드 코드 (선택)

- 서버가 클라이언트에게 코드를 전달해 기능을 확장할 수 있다.

## RESTful API

- **REST 아키텍처 스타일을 따르는 API**를 REST API(RESTful API)라고 한다.
- RESTful API의 장점
    1. **확장성**: REST는 무상태 구조이기 때문에, 요청이 서로 독립적이라 **서버를 쉽게 늘릴 수 있다**.
    2. **유연성**: REST는 클라이언트 - 서버가 서로 독립된 구조이기 때문에 **서로 독립적으로 개발 및 수정이 가능하다.**
    3. **독립성**: **특정 기술에 종속되지 않고** 서로 다른 언어, 플랫폼에서도 동일 API를 사용할 수 있다.

### RESTful API 작동 방식

![image.png](attachment:1a9e5108-b189-4f20-9e6b-1ff6941f7db5:image.png)

1. **클라이언트가 요청을 보낸다.**
    - 클라이언트 애플리케이션이 API에 작업을 수행하도록 요청을 시작하고, 이 요청은 특정 URL로 전송되며 **HTTP 메서드, Header, Request Body**를 포함한다.
2. **서버가 요청을 처리한다.**
    - 서버가 요청을 받고 검증하여 권한을 확인한다.
    - 데이터 베이스에서 데이터를 검색하거나, 새 리소스를 만들거나, 기존 리소스를 업데이트하는 등의 요청을 처리한다.
3. **서버가 응답을 보낸다.**
    - 요청을 처리한 후 서버는 클라이언트에 응답을 보낸다.
    - 응답에는 결과를 나타내는 **HTTP 상태코드와 Response Body**가 함께 포함된다.
4. **데이터를 전송한다.**
    - 클라이언트와 서버 간에 전송되는 데이터를 “**표현**”이라고 한다.
    - 이때 가장 많이 사용되는 형태는 **JSON**이며, XML, HTML과 같은 형태도 있다.

### RESTful API 설계 원칙 (Endpoint 설계)

1️⃣ **URI는 정보의 리소스를 표현해야한다.**

- `GET /members/show/1` → ❌
- `GET /members/1` → ⭕️
- 행위에 대한 표현이 아닌 **리소스를 표현하는 데 중점**을 두어야 한다.

2️⃣ **리소스에 대한 행위는 HTTP Method로 표현한다.**

- `GET /members/delete/1` → ❌   /   `DELETE /members/1` → ⭕️
- `GET /members/insert/3` → ❌   /   `POST /members/3` → ⭕️

**3️⃣ 슬래시 구분자 (”/”)는 계층 관계를 나타내는데 사용한다.**

- `GET /posts/1/comments`

**4️⃣ URI 마지막 문자로 슬래시를 포함하지 않는다.**

- `GET /members/1/`  → ❌

**5️⃣ 하이픈(-)을 이용해 URI 경로 내 단어를 구분한다.**

- 불가피하게 긴 URI 경로를 사용하게 된다면 하이픈을 사용해 가독성을 높일 수 있다.
- `GET /members/1/phone-num`

**6️⃣ 언더바(_)는 URI에 사용하지 않는다.**

- 가독성을 쉽게 헤칠 수 있다. 하이픈으로 사용하도록 한다.

**7️⃣ URI 경로에는 소문자를 사용한다.**

- 소문자 사용을 원칙으로한다. 대소문자에 따라 다른 리소스로 인식하기 때문에 주의해야한다.

**8️⃣ 파일 확장자는 URI에 포함시키지 않는다.**

- 메시지 바디 내용의 포맷을 나타내기 위한 파일 확장자를 URI에 포함시키지 않는다.
- Accept header를 사용해 포맷을 나타낸다.

## 멱등성 (Idempotency)

- **동일한 연산을 여러 번 수행해도 결과가 달라지지 않는 성질**
- “한 번 실행한 것과 여러 번 실행한 것이 같은 효과를 낸다”

## HTTP 메서드의 멱등성

- 동일한 요청을 한 번 보내는 것과 여러 번 보내는 것이 동일한 효과를 가지고, 서버의 상태도 동일하게 남는 것 = “멱등하다.”
- 멱등성은 **응답이 항상 같다는 것을 의미하는 것이 아니라**, **시스템의 상태가 불변하는 성질**을 의미한다.
- 클라이언트는 무지성으로 재시도 요청을 보내면 안되고, 멱등성을 고려하여 재시도 요청을 해야 한다.

### HTTP 메서드의 멱등 여부

1️⃣ **GET** → 멱등함 ⭕️

- GET 요청을 아무리 많이 보내도 서버의 상태는 변하지 않는다.

2️⃣ **POST** → 멱등하지 않음 ❌

- POST 요청을 보낼 때마다 리소스가 생성되어 서버의 상태가 요청에 따라 변한다.

3️⃣ **PUT** → 멱등함 ⭕️

- 대상 리소스를 항상 동일한 리소스로 대체하는 것이기 때문에 요청의 개수에 상관없다.

4️⃣ **PATCH** → 멱등하지 않음 ❌

- 만약 PATCH 요청이 나이를 한살 더하는 API 라면, 요청의 개수에 따라 값이 변한다.

5️⃣ **DELETE** → 멱등함 ⭕️

- 첫 번째 응답은 200, 두 번째 응답은 404 이지만, 리소스가 없도록 만드는 요청이므로 여러 번 호출해도 서버는 같은 상태가 유지된다.

6️⃣ **HEAD, CONNECT, OPTIONS, TRACE** → 멱등함 ⭕️

### 멱등성이 보장되지 않을 경우 문제점

- 클라이언트의 POST 요청이 서버에서 처리되었지만 일시적인 네트워크 문제로 클라이언트가 성공 응답을 받지 못한 상황에서 클라이언트가 재요청을 보내게 된다면 **중복으로 데이터가 생성**될 가능성이 있다.
    
    → 시스템의 신뢰성이 떨어지고, 중복된 데이터를 처리하기 위해 수동으로 확인해야 한다.
    
- **중복 데이터 발생**
- **시스템 신뢰성 감소**
- **운영 및 유지보수 비용 증가**

### 멱등성을 지키는 방법

- POST와 같이 멱등하지 않은 HTTP 메소드에 관해서 멱등성을 개발자가 부여해주어야 한다.
    
    **👉 멱등키를 사용한다.**
    
    ```java
    public class OrderController {
    
    		private final Map<String, OrderResponse> idempotencyStore = new ConcurrentHashMap<>();
    		
    		@PostMapping
    		public ReesponseEntity<OrderResponse> placeOrder(
    				@RequestBody OrderRequest request,
    				@RequestHeader("Idempotency-Key") String idempotencyKey
    		){
    				if(idenpotencyStore.contains(idempotencyKey)){
    						return ResponseEntity.ok(idempotencyStore.get(idempotencyKey));
    				} // 이전에 저장한 멱등키에 해당하는 응답이 있는 경우 요청을 처리하지 않고 그 값을 리턴한다..
    				
    				*...*
    				
    				OrderResponse response = new OrderResponse(...);
    				idempotencyStore.put(idempotencyKey, response);
    				return ResponseEntity.ok(response);
    		}
    ```

## HTTP Method

- 클라이언트와 서버 사이에 이루어지는 요청과 응답 데이터를 전송하는 방식
- 서버에 주어진 리소스에 **수행하길 원하는 행동, 서버가 수행해야 할 동작**을 지정하는 요청을 보내는 방법

### 1️⃣ GET

- **리소스 조회 메서드**
- 원하는 정보를 조회하기 위해서 쿼리 파라미터를 사용한다.
    - `GET /members/1?username=seoyeon`
- GET 요청시 Request Body를 이용해 데이터를 전달하는 것은 대부분의 서버가 무시하기 때문에 권장하지 않는다.
- 브라우저에서 캐싱할 수 있어, 동일한 요청에 대한 응답을 캐싱하여 빠른 응답을 받을 수 있다.
- 정적 데이터 조회 → 쿼리 파라미터 없이 리소스 경로로 단순하게 조회하는 것
- 동적 데이터 조회 → 쿼리 파라미터를 이용해 데이터를 전달하고 원하는 결과를 조회하는 것
- HTML From 전송으로 데이터를 조회할 수 있다.

### 2️⃣ POST

- **전달한 데이터를 처리 및 생성을 요청하는 메서드**
- Request Body를 통해 서버에 데이터를 전달하고, 서버는 요청 데이터를 처리하여 서버의 상태를 변경한다.
- 만약 데이터를 조회할 때, 쿼리 파라미터로 조회 데이터를 넘길 수 없는 경우 POST 메서드를 사용한다.
- **Content-Type 헤더 종류**
    - **Content-Type: application/json** → JSON, TEXT, XML 데이터 전송 시 사용
    - **Content-Type: multipart/form-data** → 파일과 같은 바이너리 데이터 전송 시 사용, 파일과 다른 정보도 함께 전달할 수 있다.
    - **Content-Type: application/x-www-form-urlencoded** → Form의 내용을 HTTP 메시지 바디를 통해서 전송한다.

### 3️⃣ PUT

- **리소스를 대체하는 메서드**
- 요청 데이터에 해당하는 리소스가 있으면 덮어쓰고, 없다면 새로 생성한다.
- Request Body를 통해 업데이트하고자 하는 데이터를 보낸다.
- 만약 일부 정보만 Request Body를 통해 데이터를 전달한다면, 기존 데이터가 완전히 대체되는 특성상 전달한 데이터 이외의 데이터는 모두 삭제된다.

### 4️⃣ PATCH

- **리소스의 일부 부분을 변경하는 메서드**
- 변경하고자 하는 데이터만 Request Body에 담아보내면, 전달된 데이터만 업데이트된다.

### 5️⃣ DELETE

- **리소스를 제거하는 메서드**
- 요청에 해당하는 리소스를 제거한다.
- 성공적으로 리소스를 삭제했다면 200(OK), 204(NO CONTENT) 상태 코드를 반환한다.
- 만약 해당 리소스가 존재하지 않는다면 404(NOT FOUND) 상태 코드를 반환한다.

### 6️⃣ HEAD

- **리소스의 메타 데이터를 조회하는 메서드**
- GET과 비슷하지만, Response Body에 담긴 데이터는 받지 않는다.
- 리소스 존재 여부를 확인하거나 파일 크기, 수정 날짜 등을 확인할 때 사용한다.

### 7️⃣ OPTIONS

- **서버가 특정 리소스에 대해 지원하는 HTTP 메서드 정보를 조회하는 메서드**
- CORS 설정을 확인하는 상황에서 사용한다.

### 8️⃣ CONNECT, TRACE

- 실무에서는 거의 사용하지 않는다.

### HTTP 메서드 속성

**Safe**

- 클라이언트의 요청이 서버의 상태를 변경하지 않는다.
- `GET`, `HEAD`, `OPTIONS`

**Idempotent**

- 동일한 요청을 여러 번 보내더라도 서버의 상태가 동일하게 유지된다.
- `GET`, `PUT`, `DELETE`, `HEAD`, `OPTIONS`

**Cacheable**

- 응답 결과를 클라이언트나 프록시 서버가 캐시할 수 있다.
- 캐시된 응답은 동일한 요청에 대해 서버에 재요청 없이 캐시된 데이터를 반환한다.
- `GET`, `HEAD`, `POST`

### HTTP 상태 코드

**1XX (정보)**: 요청을 받았으며, 프로세스를 계속 진행 중 입니다.

- **100 Continue** → 요청 메시지의 첫 부분이 서버에 도착하였으며 클라이언트는 정상적으로 계속 요청할 수 있음을 의미한다.
- **101 Switching Protocol** → Upgrade 헤더에 정의된 프로토콜을 변환하기 위해 클라이언트의 요청을 수락하였음을 의미한다.
- **102 Processing** → 요청을 처리하고 있지만 제대로 된 응답을 할 수 없는 상태임을 의미한다.

**2XX (성공)**: 요청이 성공적으로 접수, 인식, 처리되었습니다.

- **200 OK** → 요청이 성공적으로 처리되었음을 의미한다.
- **201 Created** → 요청이 성공적으로 처리되어 리소스가 추가되었음을 의미한다.
- **204 No Content** → 요청을 성공적으로 처리했지만 콘텐츠를 제공하지 않는다.

**3XX (리다이렉션)**: 요청을 완료하려면 추가 작업이 필요합니다.

- **301 Moved Permanently** → 영구적으로 콘텐츠가 이동했을 때 사용된다.
- **302 Found** → 일시적으로 콘텐츠가 이동했을 때 사용된다.
- **303 See Other** → 서버가 사용자의 GET 요청을 처리하여 다른 URl에서 요청된 정보를 가져올 수 있도록 한다.
- **307 Temporary Redirect** → 302 와 비슷하지만 메서드는 그대로 유지한다.
- **308 Permanent Redirect** → 301 와 비슷하지만 메서드는 그대로 유지된다.

**4XX (클라이언트 오류)**: 요청의 문법이 잘못되었거나 요청을 처리할 수 없습니다.

- **400 Bad Request** → 요청의 문법이 잘못되어 서버가 요청을 이해할 수 없음을 의미한다.
- **401 Unauthorized** → 인증이 필요한 리소스에 접근할 경우 발생한다. (서버가 클라이언트를 인식하지 못함)
- **403 Forbidden** → 클라이언트가 콘텐츠에 접근할 권한이 없음을 의미한다. 401과 다르게 서버가 클라이언트를 인식한 상태이다.

**5XX (서버 오류)**: 서버가 유효한 요청을 처리하는 데 실패했습니다.

- **500 Internal Server Error** → 서버에 문제가 있지만 명확한 문제를 설명할 수 없음을 의미한다.
- **502 Bad Gateway** → 서버가 게이트웨이로부터 잘못된 응답을 수신했음을 의미한다.
